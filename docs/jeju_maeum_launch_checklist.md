# 제주의마음 여행 AI 출시 전 체크리스트

자동 점검 가능한 항목은 먼저 아래 명령으로 확인한다.

```bash
python scripts/build_operations_readiness_report.py --place-cards data/jeju_accessible_spots.json --data-request-tracker data/data_request_tracker.json --service-seed-gate-status data/roadview_service_seed_gate_status.json --output data/operations_readiness_report.json --web-output web/data/operations_readiness_report.json --generated-at 2026-07-09
python scripts/build_service_preflight_report.py --workspace-root . --output data/service_preflight_report.json --output-md docs/service_preflight_report_20260709.md --generated-at 2026-07-09
```

자동 점검 결과가 `blocked_for_full_service`이거나 사전점검 결과가 `block`이면 상용 전체 공개를 보류한다. 수동 검수 항목은 아래 체크리스트를 계속 사용한다.

## 1. 데이터 검수

- 모든 장소 카드에 `id`, `name`, `region`, `category`가 있다.
- 모든 장소 카드에 최소 1개 이상의 출처가 있다.
- 모든 출처 URL이 실제로 열리는지 확인했다.
- 공식 정보와 운영자 메모를 구분했다.
- 정보 확인일이 없는 장소는 `needs_check`로 표시했다.
- 휠체어 접근성, 장애인 화장실, 주차, 경사/계단, 휴식 공간 중 빠진 항목이 표시된다.
- 접근성 정보가 부족한 장소는 높은 등급을 받지 않는다.
- 운영자가 숨긴 장소는 사용자 결과에 나오지 않는다.

## 2. 추천 결과 검수

- 사용자 조건 요약이 먼저 표시된다.
- 추천 코스는 2~4개 지점을 넘지 않는다.
- 점수와 등급이 함께 표시된다.
- 적합 이유와 감점 이유가 함께 표시된다.
- 방문 전 확인사항이 체크리스트로 표시된다.
- 출처와 정보 상태가 표시된다.
- 정보 부족 장소는 "확인 필요"로 표시된다.
- 사용자가 피하고 싶은 조건이 추천 결과에 반영된다.
- 음식 제한, 혼잡 회피, 날씨 민감 등 상황별 제약이 제외 또는 감점으로 반영된다.

## 3. 안전 문구 검수

- 모든 결과에 기본 안전 문구가 포함된다.
- 의료 진단, 치료 효과, 회복 보장 표현이 없다.
- "누구나", "무조건", "100%", "문제없이", "보장" 표현이 없다.
- 휠체어 접근 가능성을 단정하지 않는다.
- 현장 상황에 따라 접근성이 달라질 수 있음을 안내한다.
- 보호자 동행 또는 현장 문의가 필요한 경우 명확히 표시한다.

## 4. 개인정보 검수

- 이름, 연락처, 주민등록번호, 병원명, 상세 진단명을 요구하지 않는다.
- 건강 정보 입력은 이동 조건 수준으로만 유도한다.
- 사용자 입력 원문을 저장하지 않는 것이 기본값이다.
- 피드백 수집 시 별도 동의 문구가 있다.
- 운영 로그에 민감정보가 남지 않는다.

## 5. 접근성 UI 검수

- 점수 상태를 색상만으로 구분하지 않는다.
- 버튼과 입력창의 텍스트가 모바일에서 잘리지 않는다.
- 안내 문구가 너무 작지 않다.
- 키보드만으로 주요 흐름을 사용할 수 있다.
- 오류 메시지가 구체적이고 사용자가 다음 행동을 알 수 있다.
- 경고 문구는 눈에 띄지만 과도하게 불안감을 주지 않는다.

## 6. 실패 상황 검수

- 조건에 맞는 장소가 없을 때 무리하게 추천하지 않는다.
- 출처 정보가 부족할 때 정보 부족을 명확히 표시한다.
- AI 응답 파싱 실패 시 사용자에게 안전한 오류 메시지를 보여준다.
- 금지 표현이 생성되면 결과를 재작성하거나 차단한다.
- 외부 API 장애 시 기본 안내와 재시도 방법을 제공한다.
- 데이터 파일이 비어 있어도 서비스가 비정상 종료되지 않는다.

## 7. 운영 검수

- 장소 카드 추가 절차가 문서화되어 있다.
- 장소 카드 수정 절차가 문서화되어 있다.
- 정보 확인일 갱신 기준이 정해져 있다.
- 사용자 피드백을 기록할 위치가 있다.
- 부적절한 추천 사례를 기록할 위치가 있다.
- 배포 전 테스트 입력 10개를 모두 실행한다.

## 8. 출시 보류 조건

아래 항목 중 하나라도 발생하면 출시하지 않는다.

- 출처 없는 장소가 추천 결과에 노출된다.
- 안전 문구가 누락된다.
- 의료 효과를 암시하는 문장이 나온다.
- 휠체어 접근 가능성을 보장하는 문장이 나온다.
- 정보 부족 장소가 A등급으로 표시된다.
- 사용자가 피하고 싶은 조건을 가진 장소가 핵심 추천으로 나온다.
- 명시적으로 식당·음식 장소 제외를 요청했는데 식당 또는 음식 중심 시장이 추천된다.
- 개인정보가 로그에 저장된다.

## 9. 출시 가능 조건

아래 기준을 모두 만족하면 제한 공개가 가능하다.

- 출처 표시율 100%
- 안전 문구 포함률 100%
- 금지 표현 차단률 100%
- 테스트 입력 10개 통과
- 장소 카드 최소 30개 검수 완료
- 정보 상태 `verified` 또는 `partial` 장소가 전체의 70% 이상
- `needs_check` 장소는 결과에서 명확히 확인 필요로 표시
