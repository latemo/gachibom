# 제주의마음 여행 AI 서비스화 계획

## 1. 방향 전환

이 프로젝트는 데모용 추천기가 아니라 실제 사용자가 여행 전에 참고할 수 있는 서비스로 설계한다.

따라서 핵심 기준은 "작동하는 화면"이 아니라 "사용자가 믿고 확인할 수 있는 정보 품질"이다.

서비스화 기준:

- 접근성 정보의 출처가 명확해야 한다.
- 장소별 정보의 최신성, 확인 필요 여부, 불확실성을 표시해야 한다.
- 사용자 건강 상태를 진단하거나 여행 가능성을 보장하지 않아야 한다.
- 추천보다 확인과 판단 보조에 초점을 둬야 한다.
- 운영자가 장소 카드와 주의 문구를 계속 갱신할 수 있어야 한다.

## 2. 서비스 포지셔닝

제주의마음 여행 AI는 제주 관광약자를 위한 무장애 여행 판단 보조 서비스다.

서비스가 하는 일:

- 사용자의 이동 조건을 정리한다.
- 음식 제한, 혼잡 회피, 날씨 민감 등 여행 제약을 정리한다.
- 제주 접근성 정보를 검색한다.
- 장소와 코스의 적합도를 설명한다.
- 위험하거나 확인이 필요한 요소를 알려준다.
- 보호자가 전화나 공식 페이지로 확인할 항목을 체크리스트화한다.

서비스가 하지 않는 일:

- 의료 판단
- 치료 효과 안내
- 사용자가 먹어도 되는 음식 판단
- 이동 가능성 보장
- 실시간 현장 상태 보장
- 장애 정도나 건강 상태의 자동 판정
- 무조건적인 관광지 추천

## 3. 제품 원칙

1. 추천보다 근거
   - 모든 결과에는 장소 카드, 출처, 확인 필요 항목이 함께 나와야 한다.

2. 가능보다 부담
   - "갈 수 있다"보다 "얼마나 무리 없는가"를 중심으로 판단한다.

3. 자동화보다 운영 가능성
   - 초기에는 운영자가 검수한 접근성 카드 기반으로 시작한다.

4. 감성보다 안전
   - 따뜻한 문체는 유지하되, 위험 요소와 한계를 흐리지 않는다.

5. 색상보다 텍스트
   - 접근성 점수와 상태는 색상만이 아니라 점수, 등급, 문장으로 함께 표시한다.

## 4. 1차 출시 범위

1차 출시는 "제주 무장애 여행 사전 확인 서비스"로 잡는다.

필수 범위:

- 자연어 여행 조건 입력
- 여행자 조건 요약
- 접근성 장소 후보 검색
- 코스 적합도 점수
- 추천 이유와 감점 이유
- 방문 전 확인 체크리스트
- 출처 표시
- 안전 안내 문구
- 운영자용 장소 카드 데이터 구조

제외 범위:

- 회원가입
- 결제
- 예약
- 실시간 교통
- 실시간 혼잡도
- 의료 상담
- 병원 연계
- 개인 건강정보 장기 저장

## 5. 사용자 흐름

1. 사용자가 여행자의 상태와 원하는 제주 경험을 입력한다.
2. 서비스가 이동 조건, 선호 테마, 필수 편의시설, 피해야 할 요소를 요약한다.
3. 사용자가 요약을 확인하고 필요하면 수정한다.
4. 서비스가 접근성 장소 카드를 검색한다.
5. 후보 장소별 적합도와 불확실성을 계산한다.
6. 2~4개 지점의 무리 없는 코스를 제안한다.
7. 방문 전 확인 체크리스트와 출처를 보여준다.
8. 사용자는 공식 페이지나 전화로 최종 확인한다.

## 6. 데이터 운영 계획

서비스 품질의 핵심은 장소 카드다.

장소 카드에는 최소한 다음 정보가 있어야 한다.

- 장소명
- 지역
- 카테고리
- 추천 사용자군
- 피해야 할 사용자군
- 접근성 태그
- 주차 정보
- 장애인 화장실 정보
- 경사로 또는 계단 정보
- 예상 도보 부담
- 휴식 가능성
- 휠체어 대여 또는 보조기기 관련 정보
- 확인 필요 항목
- 출처명
- 출처 URL
- 정보 확인일
- 운영자 메모

데이터 운영 방식:

- 초기 30개 장소는 사람이 직접 검수한다.
- 각 장소는 "확인됨", "부분 확인", "확인 필요" 상태를 가진다.
- 정보 확인일이 오래된 장소는 결과에서 신뢰도를 낮춘다.
- 출처가 불명확한 정보는 추천 근거로 쓰지 않는다.
- 후기성 문구는 참고로만 사용하고 공식 근거와 분리한다.

## 7. 신뢰도 표시 방식

사용자에게 단순 점수만 보여주면 안 된다.

각 결과는 다음 구조로 보여준다.

- 총점: 87점
- 등급: A
- 적합 이유: 짧은 이동, 바다 선호 일치, 주차 확인 가능
- 감점 이유: 화장실 운영 여부 현장 확인 필요
- 확인 필요: 주차장-입구 거리, 경사로 상태, 바람 강도
- 출처: 공식 페이지 또는 운영자 검수 카드
- 정보 상태: 확인됨 / 부분 확인 / 확인 필요

## 8. 안전 정책

모든 결과에는 안전 문구가 포함되어야 한다.

기본 문구:

> 이 서비스는 의료 판단이나 여행 가능성을 보장하지 않습니다. 현장 접근성은 날씨, 운영 상황, 공사, 혼잡도에 따라 달라질 수 있으므로 방문 전 공식 정보와 현장 문의를 확인해 주세요.

금지해야 할 표현:

- 누구나 갈 수 있습니다.
- 휠체어도 100% 가능합니다.
- 암환자에게 좋습니다.
- 치료에 도움이 됩니다.
- 무조건 추천합니다.
- 문제없이 이동할 수 있습니다.

권장 표현:

- 방문 전 확인이 필요합니다.
- 이동 부담이 낮은 편으로 판단됩니다.
- 보호자 동행 기준으로 무리가 적은 코스입니다.
- 현장 상황에 따라 접근성이 달라질 수 있습니다.
- 장시간 체류보다 짧은 감상 중심으로 권장합니다.

## 9. 운영자 기능

서비스화하려면 사용자 화면만으로는 부족하다.

운영자에게 필요한 기능:

- 장소 카드 추가
- 장소 카드 수정
- 출처 URL 관리
- 정보 확인일 업데이트
- 확인 필요 항목 표시
- 부적절한 추천 결과 로그 확인
- 사용자 피드백 수집
- 위험 문구 필터 관리

초기에는 별도 관리자 페이지 없이 JSON, CSV, 또는 스프레드시트로 운영해도 된다. 단, 데이터 구조는 나중에 관리자 페이지로 이전 가능해야 한다.

## 10. 출시 전 검증 기준

서비스로 내기 전 최소 검증 기준을 둔다.

필수 기준:

- 모든 결과에 출처 또는 근거 카드가 표시된다.
- 모든 결과에 안전 문구가 표시된다.
- 이동 부담이 큰 장소는 높은 점수를 받지 않는다.
- 사용자가 피하고 싶은 요소가 추천 이유와 감점 이유에 반영된다.
- 접근성 정보가 부족한 장소는 "확인 필요"로 표시된다.
- 금지 표현이 결과에 나오지 않는다.

정량 기준:

- 출처 표시율: 100%
- 안전 문구 포함률: 100%
- 금지 표현 차단률: 100%
- 사용자 조건 반영률: 90% 이상 목표
- 확인 필요 항목 표시율: 90% 이상 목표

## 11. 단계별 로드맵

### 1단계. 서비스 기획 고정

- 대표 사용자군 확정
- 결과 화면 구조 확정
- 장소 카드 스키마 확정
- 안전 문구와 금지 표현 확정

### 2단계. 데이터 구축

- 제주 접근성 장소 30개 카드 작성
- 출처 URL 정리
- 확인 필요 항목 정리
- 테스트 입력 20개 작성

### 3단계. 1차 서비스 구현

- 사용자 입력 화면
- 조건 분석 화면
- 추천 결과 화면
- 체크리스트 화면
- 출처와 안전 안내 화면
- 운영자 데이터 파일 로딩

### 4단계. 품질 검증

- 사용자군별 테스트
- 금지 표현 테스트
- 접근성 정보 부족 상황 테스트
- 점수 과대평가 테스트
- 보호자 관점 문장 검토

### 5단계. 제한 공개

- 지인, 복지기관 담당자, 보호자 대상 피드백 수집
- 잘못된 추천 사례 기록
- 장소 카드 보강
- 화면 문구 개선

## 12. 다음 작업

우선 만들어야 할 산출물:

- 장소 카드 JSON 스키마
- 샘플 장소 카드 5개
- 사용자 테스트 입력 10개

- 금지 표현 목록
- 방문 전 체크리스트 템플릿
