# 제주의마음 장소 데이터 운영 기준

## 목적

제주의마음은 데모용 장소 목록이 아니라 실제 서비스에서 계속 늘어나는 제주 장소 데이터를 다룬다. 따라서 모든 장소를 바로 추천하지 않고, 원본 장소 목록, 접근성 검증 카드, 사람 검수 상태를 분리해 운영한다.

이 문서는 장소를 어떻게 수집하고, 어떤 조건에서 서비스에 노출하며, 상황별 추천 점수에 어떤 근거로 반영하는지 정리한다.

## 현재 데이터 상태

기본 접근성 장소 카드는 `data/jeju_accessible_spots.json`에서 관리한다.

현재 기준:

- 전체 장소 카드: 43곳
- 서비스 기본 추천 후보: `active`이면서 `verified` 또는 `partial`인 장소
- 검수 보류 후보: `needs_check` 또는 `unavailable`인 장소
- 로드뷰 시설 원본 카탈로그: 107곳
- 전체 카탈로그 확장 대상: 관광지, 식당, 카페, 숙박, 쇼핑, 문화시설, 행사, 체험, 교통 거점, 의료지원 참고 지점

자동 요약 산출물은 아래 명령으로 다시 만든다.

```powershell
python scripts/build_place_data_operations_summary.py `
  --raw-catalog data/place_catalog.roadview_facility.json `
  --output data/place_data_operations_summary.json `
  --generated-at 2026-07-09
```

## 데이터 레이어

### 원본 장소 카탈로그

제주 전체 장소의 존재와 기본 정보를 담는다.

- 장소명
- 분류
- 주소와 좌표
- 연락처
- 원본 출처
- 원본 데이터 갱신일

이 레이어에 있다는 것만으로 접근 가능하다는 뜻은 아니다.

현재 로드뷰 시설 원본 카탈로그는 `data/place_catalog.roadview_facility.json`에 저장한다. 이 파일은 107개 관광지의 기본 정보와 기존 접근성 카드 매칭 상태를 담는다.

### 접근성 장소 카드

추천 점수 계산에 직접 쓰는 운영 카드다.

- 휠체어 접근
- 장애인 화장실
- 주차
- 경사·계단
- 휴식 공간
- 대여·보조
- 바닥 상태
- 혼잡도
- 도보 부담
- 야외 노출
- 날씨 민감도
- 출처
- 안전 메모
- 검증 상태

### 매칭과 검수 레이어

원본 장소와 접근성 장소 카드를 연결한다.

- `matched`: 원본 장소와 접근성 카드가 확실히 연결됨
- `candidate`: 이름·지역이 비슷해 검수 후보
- `unmatched`: 아직 접근성 카드 없음
- `manual_review`: 사람이 확인해야 함

## 서비스 노출 기준

### 기본 추천 후보

아래 조건을 모두 만족해야 기본 추천 결과에 들어갈 수 있다.

- `status`가 `active`
- `verification.status`가 `verified` 또는 `partial`
- 공식 또는 준공식 출처가 1개 이상 있음
- 방문 전 주의사항 또는 안전 메모가 있음
- 사용자 상황별 제외 규칙에 걸리지 않음

### 검수 전용 후보

아래 상태는 데이터에는 보관하지만 기본 추천에는 넣지 않는다.

- `verification.status`가 `needs_check` 또는 `unavailable`
- 접근성 필수 필드가 비어 있음
- 신규 카테고리라 추천 기준이 아직 없음
- 로드뷰 이미지 또는 현장 근거가 사람 검수 전 상태

### 노출 금지 후보

아래 상태는 사용자 추천 결과에 노출하지 않는다.

- `status`가 `hidden`, `blocked`, `deprecated`
- 사용자 회피 조건과 장소 주의사항이 강하게 충돌
- 음식 제한 조건에서 식당 또는 음식 중심 시장
- 의료적 효과를 이유로 추천해야만 하는 요청

## 상태값 기준

### 운영 상태

- `active`: 추천 후보로 계산 가능. 단, 상황별 제외와 검증 상태를 다시 적용한다.
- `hidden`: 내부 보존용. 사용자 화면에는 노출하지 않는다.
- `blocked`: 운영상 부적합하거나 오류가 있어 노출하지 않는다.
- `deprecated`: 폐업, 이전, 대체 등으로 주 추천 데이터에서 제외한다.

### 검증 상태

- `verified`: 출처와 핵심 접근성 필드가 충분해 기본 추천 후보로 사용 가능하다.
- `partial`: 추천 후보로 사용 가능하지만 화면에 확인 필요 문구와 근거를 함께 표시한다.
- `needs_check`: 기본 추천에서 제외하고 검수 큐로 보낸다.
- `unavailable`: 접근성 판단 정보가 거의 없어 추천에서 제외한다.

## 카테고리 운영 기준

장소 카테고리는 관광지만으로 제한하지 않는다.

- `sea`: 바다, 해안, 전망 장소. 날씨와 강풍 민감 조건에서 감점 또는 확인 필요.
- `forest`: 숲, 수목원, 자연휴양림. 도보 부담, 바닥 상태, 날씨 영향을 함께 확인.
- `oreum`: 오름. 경사와 장거리 보행 가능성이 커 기본 추천 전 정밀 검수 필요.
- `culture`: 역사, 문화, 공연 공간. 실내외 동선과 화장실 접근성 확인.
- `indoor`: 실내 전시, 박물관, 기념관. 회복 중·날씨 민감 조건에서 우선 후보.
- `cafe`: 카페, 차, 디저트 장소. 음식 제한·혼잡 민감 조건에서 감점.
- `restaurant`: 식당. 음식 제한 조건에서는 기본 제외.
- `food_market`: 시장, 야시장, 음식 중심 상권. 음식 제한 조건에서는 제외, 혼잡 민감 조건에서는 감점.
- `shopping`: 쇼핑 중심 장소. 혼잡 민감 조건에서 감점.
- `rest_area`: 공원, 휴식 공간. 휴식 가능 여부와 야외 노출을 함께 확인.
- `transport`: 공항, 터미널 등 교통 거점. 여행 보조 정보로 사용.
- `medical_support`: 병원, 응급 대응 참고 지점. 추천 여행지가 아니라 지원 정보로 분리.
- `lodging`: 숙박. 접근 객실, 욕실, 주차, 엘리베이터 검수 기준을 별도로 둔다.
- `event`: 축제와 행사. 기간, 혼잡도, 임시 동선 검수 전까지 기본 추천에서 제외한다.
- `experience`: 체험 관광. 프로그램별 접근성 검수 후 노출한다.

## 상황별 추천 반영

상황 규칙은 `data/situation_rules.json`에서 관리한다.

예시:

- 음식 제한: `restaurant`, `food_market` 제외, `cafe` 감점
- 혼잡·감염 민감: 시장, 쇼핑, 카페, 혼잡 가능 장소 감점
- 체력 저하: 긴 걷기, 높은 야외 노출, 휴식 부족 장소 감점
- 날씨 민감: 해안, 오름, 장시간 야외 장소 감점
- 감각 민감: 강한 조명, 큰 소리, 어두운 전시 공간 확인 필요

이 규칙은 진단명이 아니라 사용자가 입력한 여행 제약을 처리한다.

## 추천 점수와 집계 근거

점수 계산은 `src/scoring.py`에서 처리한다.

총점은 100점이며 아래 5개 축으로 계산한다.

- 출처 신뢰도: 25점
- 이동 적합도: 25점
- 시설 적합도: 20점
- 테마 적합도: 15점
- 안전 명확성: 15점

점수는 장소 하나마다 먼저 계산하고, 코스 점수는 선택된 장소들의 평균으로 집계한다. 정보가 부족하거나 사용자 조건과 충돌하면 강제 감점 또는 차단을 적용한다.

주요 차단 기준:

- 운영 상태가 `hidden`, `blocked`, `deprecated`
- 검증 상태가 `unavailable`
- 음식 제한 조건에서 식당 또는 음식 중심 시장
- 사용자가 피하고 싶은 조건과 장소 주의사항이 충돌
- 의료적 효과를 보장하는 방식의 추천 요청

검증 상태에 따른 상한도 적용한다.

- `needs_check`: 최대 84점
- `unavailable`: 최대 69점

따라서 점수가 높더라도 검수 상태가 부족하면 화면에서는 낮은 신뢰도나 확인 필요로 표시한다.

## 확장 운영 순서

1. 공공데이터 또는 공식 출처에서 원본 장소를 수집한다.
2. `raw catalog`로 먼저 저장한다.
3. 기존 접근성 카드와 자동 매칭한다.
4. 매칭되지 않은 장소는 검수 큐로 보낸다.
5. 접근성 필드, 출처, 안전 메모를 보강한다.
6. `needs_check`에서 `partial` 또는 `verified`로 승격한다.
7. 상황별 추천 규칙과 점수 로직으로 노출 여부를 다시 계산한다.
8. 앱 화면에는 출처, 확인 필요 항목, 안전 문구를 함께 표시한다.
