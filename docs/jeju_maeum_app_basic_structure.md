# 제주의마음 앱 기본구조

작성일: 2026-07-08

## 1. 앱의 역할

제주의마음 앱은 관광지를 많이 보여주는 서비스가 아니라, 사용자의 이동 조건과 제주 접근성 데이터를 대조해 무리 없는 장소와 코스를 제안하는 서비스다.

앱은 세 가지 질문에 답해야 한다.

- 이 사용자에게 추천해도 되는 장소인가
- 추천한다면 어떤 근거와 점수로 추천하는가
- 방문 전에 무엇을 확인해야 하는가

의료 판단, 여행 가능성 보장, 실시간 현장 보장은 하지 않는다.

## 2. 1차 앱 범위

1차 앱은 회원가입이나 예약 기능 없이 추천 흐름에 집중한다.

- 사용자 조건 입력
- 조건 분석 요약
- 추천 장소와 코스 생성
- 점수, 추천 이유, 감점 이유 표시
- 방문 전 확인 항목 표시
- 출처와 안전 안내 표시

제외 기능:

- 회원가입
- 개인 건강 기록 저장
- 실시간 예약
- 실시간 교통 최적화
- 의료 상담
- 검수 미완료 장소의 사용자 노출

## 3. 핵심 화면

### 화면 1. 추천 조건 입력

사용자가 자신의 상황을 빠르게 선택한다.

필수 입력:

- 여행자 상황: 휠체어, 회복 중, 고령자, 유모차, 보호자 동반
- 이동 조건: 긴 걷기 어려움, 경사 민감, 계단 회피, 짧은 체류
- 선호 테마: 바다, 숲, 실내, 문화, 조용한 장소
- 필수 편의시설: 장애인 화장실, 가까운 주차, 휴식 공간, 대여 정보
- 제외 조건: 식당 제외, 혼잡 회피, 장시간 야외 회피, 강한 자극 회피

자연어 입력은 보조 수단으로 둔다. 1차 화면은 선택형 입력을 우선한다.

### 화면 2. 조건 분석 요약

앱이 해석한 조건을 짧게 보여준다.

- 여행자 유형
- 이동 부담 기준
- 필수 확인 항목
- 제외 또는 감점 규칙
- 추천 신뢰도 기준

이 화면은 별도 단계로 길게 만들지 않고, 추천 결과 상단에 항상 붙인다.

### 화면 3. 추천 결과

추천 결과는 장소 카드가 아니라 판단 리포트에 가깝게 보여준다.

각 장소는 다음 정보를 가진다.

- 장소명
- 적합도 점수와 등급
- 추천 이유
- 감점 이유
- 방문 전 확인 항목
- 정보 신뢰도
- 출처 요약

사용자에게는 높은 점수만 보여주지 않고, 왜 감점됐는지 함께 보여준다.

### 화면 4. 코스 보기

코스는 2~4개 장소로 제한한다.

- 코스 제목
- 이동 속도: 매우 천천히, 천천히
- 장소 순서
- 각 장소의 목적
- 쉬는 지점 또는 확인 지점
- 코스 전체 점수

조건에 맞는 장소가 부족하면 무리하게 코스를 만들지 않고 추천 보류를 표시한다.

### 화면 5. 장소 상세

장소 상세는 접근성 근거 중심으로 구성한다.

- 접근성 요약
- 화장실, 주차, 경사, 바닥, 휴식, 대여 정보
- 로드뷰 검수 상태
- 방문 전 전화 또는 공식 정보 확인 항목
- 출처와 확인일

## 4. 추천 처리 흐름

앱 추천 흐름은 다음 순서로 고정한다.

1. 사용자 조건을 구조화한다.
2. 공개 가능한 장소만 후보로 만든다.
3. 사용자 제외 조건을 먼저 적용한다.
4. 남은 후보에 점수를 계산한다.
5. 차단 장소는 기본 추천에서 제외한다.
6. 점수, 등급, 신뢰도 순으로 정렬한다.
7. 상위 장소를 2~4개 코스로 묶는다.
8. 추천 이유, 감점 이유, 확인 항목, 출처를 함께 보여준다.

현재 구현 기준:

- 점수 계산: `src/scoring.py`
- 점수 정책: `docs/jeju_maeum_scoring_policy.md`
- 상황별 제외와 감점: `data/situation_rules.json`
- 장소 데이터: `data/jeju_accessible_spots.json`
- 추천 결과 스키마: `data/schemas/recommendation_result.schema.json`

## 5. 데이터 공개 게이트

사용자 앱은 검수 완료 또는 제한 공개 가능한 데이터만 사용한다.

기본 제외:

- `hidden` 상태 장소
- `blocked` 상태 장소
- `deprecated` 상태 장소
- 로드뷰 시각 검수에서 사람 최종 판정이 없는 서비스 시드 장소
- 출처 또는 안전 메모가 부족한 장소

로드뷰 기반 신규 장소는 운영자 검수가 끝나기 전까지 사용자 추천 결과에 넣지 않는다.

## 6. 화면 상태

앱에는 최소 다섯 가지 결과 상태가 필요하다.

| 상태 | 의미 | 화면 처리 |
| --- | --- | --- |
| 추천 가능 | 조건과 맞는 장소가 있음 | 추천 장소와 코스 표시 |
| 조건부 가능 | 일부 확인 필요 | 점수와 확인 항목 강조 |
| 추천 보류 | 조건에 맞는 장소 부족 | 무리한 추천 대신 확인 안내 |
| 데이터 부족 | 필수 접근성 정보 부족 | 공개 추천 제외 |
| 검수 대기 | 운영자 검수 미완료 | 사용자 화면 미노출 |

## 7. 1차 시안 기준

첫 시안은 랜딩페이지가 아니라 실제 추천 화면이어야 한다.

첫 화면 구성:

- 왼쪽: 사용자 조건 입력
- 오른쪽: 조건 분석 요약과 추천 결과
- 하단: 방문 전 확인 항목과 안전 문구

시안 파일:

- `docs/jeju_maeum_app_recommendation_screen.html`

## 8. 다음 구현 순서

1. 정적 시안 확정
2. 실제 장소 데이터 43개를 화면용 샘플 데이터로 연결
3. `src/scoring.py` 결과를 화면 데이터 형식으로 변환하는 어댑터 작성
4. 추천 결과 화면을 실제 점수 결과로 렌더링
5. 장소 상세 화면 추가
6. 검수 게이트를 앱 데이터 로딩 단계에 연결

## 9. 현재 앱 뼈대

현재 앱 뼈대는 새 프론트엔드 패키지 없이 `web` 폴더에 정적 앱으로 구성했다.

- 실행 화면: `web/index.html`
- 화면 스타일: `web/styles.css`
- 화면 동작: `web/app.js`
- 앱 데이터: `web/data/app_recommendation_seed.json`
- 데이터 생성: `scripts/build_app_recommendation_seed.py`
- 데이터 생성 로직: `src/app_recommendations.py`

이 구조는 기술스택을 아직 고정하지 않고도 실제 추천 데이터 연결을 검증하기 위한 단계다. 이후 리액트, 네이티브 앱, 서버 렌더링 구조를 선택하더라도 앱이 읽는 데이터 경계는 유지할 수 있다.
