# 제주의마음 여행 AI 최종 통합 상용화 기획안

## 1. 문서 목적

이 문서는 새로 받은 `jeju_maeum_final_commercialization_plan.docx`의 상용화 관점과, 현재 프로젝트에 정리된 서비스 기준서·데이터 구조·점수화 정책·출시 기준을 통합한 최종 기준 문서다.

문서의 역할은 둘로 나뉜다.

- 외부 설명용: 투자, 제휴, 공공 협력, 발표에서 사용할 상위 사업계획
- 내부 실행용: 실제 서비스 구현, 데이터 운영, AI 응답, 출시 검수에서 지켜야 할 기준

새 GPT 기획안은 상위 사업계획의 구조로 채택한다. 단, 실제 제품 기준은 현재 프로젝트의 안전·데이터·점수화 정책을 우선한다.

## 2. 최종 한 문장 정의

제주의마음 여행 AI는 관광약자의 이동 조건과 제주 접근성 정보를 대조해, 무리 없는 여행 코스와 방문 전 확인사항을 제공하는 제주 무장애 여행 검증·계획·운영 플랫폼이다.

핵심은 인기 관광지를 많이 보여주는 것이 아니다.

핵심은 이 사용자의 조건에서 실제로 무엇을 확인해야 하는지, 어떤 장소가 상대적으로 부담이 낮은지, 어떤 경우에는 제외해야 하는지를 근거와 함께 설명하는 것이다.

## 3. 최종 포지셔닝

### 외부 포지셔닝

관광약자 맞춤 제주 무장애 여행 검증·계획·운영 플랫폼.

### 내부 제품 원칙

제주의마음 여행 AI는 추천 서비스이기 전에 판단 보조 서비스다.

서비스는 다음을 약속하지 않는다.

- 여행 가능성 보장
- 휠체어 접근성 100% 보장
- 실시간 현장 상태 보장
- 의료 상담, 진단, 치료 효과 판단
- 사용자가 먹어도 되는 음식 판단

서비스가 반드시 제공해야 하는 것은 다음이다.

- 사용자 이동 조건 요약
- 장소별 접근성 근거
- 정보 신뢰도와 사용자 적합도
- 감점 이유와 제외 이유
- 방문 전 확인 체크리스트
- 출처, 확인일, 정보 상태
- 안전 안내 문구

## 7. AI·RAG·점수화 원칙

AI는 최종 판단을 단독으로 하지 않는다.

| 역할 | 담당 |
| --- | --- |
| 사용자 입력 구조화 | LLM |
| 검색 질의 생성 | LLM |
| 공식·검증 데이터 검색 | RAG |
| 점수 산정 | 룰 기반 scoring engine |
| 감점·제외 처리 | 룰 기반 scoring engine |
| 설명 문장 생성 | LLM |
| 금지 표현 제거 | 안전 필터와 정책 |

점수는 LLM이 임의로 만들지 않는다. 점수는 공식 출처, 운영자 검수, 접근성 필드, 사용자 조건, 상황 규칙, 정보 최신성을 기반으로 계산한다.

### 점수 구조

| 평가 항목 | 배점 | 의미 |
| --- | ---: | --- |
| 정보 신뢰도 | 25 | 공식 출처, 운영자 검수, 확인일, 핵심 항목 충족 |
| 이동 부담 적합성 | 25 | 도보 부담, 계단, 경사, 주차장 거리, 체류 부담 |
| 편의시설 충족도 | 20 | 장애인 화장실, 주차장, 휴식 공간, 대여 정보 |
| 선호 테마 적합성 | 15 | 바다, 숲, 실내, 문화, 조용한 장소 등 |
| 안전 안내 명확성 | 15 | 주의사항, 확인 필요 항목, 보호자 메시지 |

### 등급 기준

| 점수 | 등급 | 의미 |
| ---: | --- | --- |
| 85~100 | A | 매우 적합. 그래도 방문 전 확인 필요 |
| 70~84 | B | 대체로 적합. 일부 항목 확인 필요 |
| 50~69 | C | 조건부 가능. 보호자 확인과 대체안 필요 |
| 30~49 | D | 이동 부담 또는 정보 부족이 큼 |
| 0~29 | F | 현재 조건에서는 추천 부적합 |

### 강제 제한

- `needs_check` 장소는 최대 B등급으로 제한한다.
- `unavailable` 장소는 최대 C등급으로 제한한다.
- 출처가 없거나 접근성 판단 정보가 거의 없는 장소는 핵심 추천 장소로 쓰지 않는다.
- 사용자가 명시적으로 제외한 조건에 해당하면 추천 후보에서 제외한다.
- 식당 제외 요청이 있으면 `restaurant`, `food_market`은 제외하고 `cafe`는 감점한다.

## 8. 상황별 추천 정책

상황은 진단명이 아니라 여행 제약으로 처리한다.

금지 표현:

- 암환자에게 좋다
- 치료에 도움이 된다
- 이 음식은 먹어도 된다
- 이 코스는 안전하다

허용 표현:

- 음식 제한을 입력했으므로 식사 중심 장소는 제외합니다.
- 체력 부담을 줄이기 위해 짧은 실내 관람지를 우선합니다.
- 강풍에 민감하다고 입력했으므로 해안 야외 장소는 감점합니다.
- 혼잡을 피하고 싶다고 입력했으므로 시장과 인기 카페는 확인 필요로 표시합니다.

### 1차 상황 유형

| 상황 | 처리 |
| --- | --- |
| 음식 제한 | 식당·푸드마켓 제외, 카페 감점 |
| 감염·혼잡 민감 | 시장·인기 카페·혼잡 가능 장소 감점 |
| 체력 저하 | 장거리·긴 야외 체류 감점, 짧은 코스 우선 |
| 날씨 민감 | 해안·오름·긴 야외 장소 감점 |
| 감각 민감 | 미디어아트·야간·강한 자극 장소 확인 필요 |

## 9. 데이터 전략

전체 제주 장소를 서비스에 넣을 수는 있다. 그러나 전체 장소가 접근 가능한 장소라는 뜻은 아니다.

따라서 데이터는 세 계층으로 분리한다.

### raw place catalog

제주 전체 장소 원본 데이터다.

포함 대상:

- 관광지
- 음식점
- 카페
- 숙박
- 쇼핑
- 문화시설
- 축제·행사
- 체험관광
- 교통 거점
- 공항, 병원, 약국 같은 지원 지점

이 레이어는 장소의 존재와 기본 정보만 의미한다. 접근 가능하다는 의미가 아니다.

### accessibility place cards

운영자가 공식·공공 출처 또는 현장 검증으로 확인한 접근성 카드다.

필수 항목:

- 휠체어 접근성
- 장애인 화장실
- 주차와 장애인 전용 주차구역
- 경사, 계단, 단차
- 휴식 공간
- 휠체어 또는 유아차 대여
- 바닥 상태
- 혼잡도 또는 혼잡 확인 필요 여부
- 출처 URL
- 정보 확인일
- 안전 메모

### matching layer

원본 장소와 접근성 카드를 연결한다.

| 상태 | 의미 |
| --- | --- |
| `matched` | 접근성 카드와 확실히 연결됨 |
| `candidate` | 이름이나 지역이 비슷해 검수 후보 |
| `unmatched` | 아직 접근성 검증 없음 |
| `manual_review` | 사람이 확인해야 함 |

`unmatched` 장소는 추천 결과에서 정보 부족 또는 확인 필요로 표시한다. 핵심 추천 장소로 올릴 때는 강한 제한을 둔다.

## 10. 데이터 수집과 import 운영

우선 출처:

- 이지제주 EASYJEJU
- VISIT JEJU 모두를 위한 제주
- 열린관광 모두의 여행
- 제주특별자치도, 제주관광공사, 한국관광공사 등 공공기관 자료
- 장소 공식 홈페이지

후기, 블로그, SNS는 현장 변화 가능성과 사진 참고용으로만 사용한다. 추천 근거로 직접 쓰지 않는다.

### 현재 구현된 import 구조

현재 프로젝트에는 실제 공공데이터를 내부 카탈로그로 넣기 위한 구조가 있다.

- `src/catalog.py`: 장소 카탈로그 import와 접근성 카드 매칭
- `src/catalog_providers.py`: 공공데이터/VISIT JEJU 필드 정규화
- `src/json_records.py`: API JSON record 추출과 중첩 필드 평탄화
- `src/roadview_data.py`: 사회적약자 시설현황과 로드뷰 이미지 메타데이터 변환
- `src/roadview_merge.py`: roadview 접근성 카드 초안과 기존 카드의 병합 검수 리포트 생성 및 보수적 자동 반영
- `scripts/import_place_catalog.py`: CSV import CLI
- `scripts/import_place_catalog_json.py`: JSON/API import CLI
- `scripts/import_roadview_facility_cards.py`: 107개 관광지 시설현황 CSV를 접근성 카드 초안과 카탈로그로 변환
- `scripts/import_roadview_metadata.py`: 4,748건 로드뷰 이미지 메타데이터 CSV를 보조 JSON으로 변환
- `scripts/review_roadview_merge.py`: 기존 접근성 카드와 roadview 초안을 매칭·신규·수동검수·보강가능 항목으로 분류
- `scripts/apply_roadview_safe_updates.py`: 충돌 없는 `yes` 근거만 기존 접근성 카드에 반영하고 나머지는 수동 검수 큐로 분리
- `scripts/apply_roadview_conflict_resolutions.py`: 운영자가 확정한 충돌 해소 판단을 카드 운영 메모와 오픈 검수 큐에 반영
- `scripts/apply_roadview_match_resolutions.py`: 운영자가 확정한 불확실 매칭 판단을 카드 운영 메모와 오픈 검수 큐에 반영
- `scripts/triage_roadview_new_candidates.py`: 신규 후보를 서비스 시드 후보, 카탈로그 후보, 현장검수 우선 후보로 분류
- `scripts/review_service_seed_cards.py`: 서비스 시드 후보의 공개 전 차단 사유와 로드뷰 이미지 검수 가능 여부를 리포트화
- `scripts/build_service_seed_work_queue.py`: 공식 출처, 로드뷰 이미지, 혼잡도, 카테고리 보정 작업 큐 생성

운영 원칙:

- 원본 API 응답은 `data/raw/` 아래 날짜 포함 파일명으로 보존한다.
- API 키는 코드, 문서, 채팅에 적지 않는다.
- API 키는 환경변수로만 전달한다.
- import 산출물은 `place_catalog` JSON이다.
- 접근성 검증은 별도 카드와의 매칭 결과로만 판단한다.

사회적약자 시설현황 데이터 사용 기준:

- 107개 관광지 시설현황은 접근성 카드 초안과 기존 카드 보강에 사용한다.
- 장애인 화장실 수, 장애인 주차장 수, 휠체어 대여, 수유실, 휴게실은 공공데이터 근거로 반영한다.
- 경사, 단차, 바닥 상태, 혼잡도, 현재 운영 여부는 자동 확정하지 않는다.
- 4,748장 로드뷰 이미지는 repository에 저장하지 않고 Object Storage에 분리 보관한다.
- 로드뷰 이미지 메타데이터는 운영자 검수 화면에서 근거 이미지 연결에 사용한다.
- roadview 초안은 바로 병합하지 않고 `roadview_merge_report`를 거친다. 자동 반영은 충돌 없는 `yes` 근거로 제한하고, 신규 후보·충돌·`no` 근거는 수동 검수 큐에서 처리한다.
- 수동 충돌 해소는 판단 파일과 적용 리포트를 남겨, 어떤 출처를 우선했는지 추적 가능하게 관리한다.
- 불확실 매칭은 동일 장소, 같은 접근성 범위, 하위 지점 포함 여부를 판단 파일로 남겨 중복 장소 생성을 방지한다.
- 신규 후보는 바로 공개하지 않고 `status=hidden` 검수용 서비스 시드 후보로 분리해 공식 상세 출처와 로드뷰 이미지 검수를 거친다.
- 서비스 시드 후보는 공식 상세 출처, 경사·단차, 바닥 상태, 혼잡도 정책 차단 사유가 해소될 때까지 공개하지 않는다.
- 공개 전 작업 큐는 공식 출처 검수와 로드뷰 이미지 검수를 high 우선순위로 처리한다.

## 12. 상용 시스템 아키텍처

기술스택은 아직 확정하지 않는다. 다만 상용 시스템은 아래 모듈 경계를 기준으로 설계한다.

```text
사용자 웹/앱
  -> API 서버 / 인증 / 계정 / 결제
  -> AI 조건 분석 서비스
  -> RAG 검색 엔진
  -> 접근성 데이터베이스 / 리뷰 / 현장 검증 데이터
  -> 신뢰도·적합도·위험도 점수 엔진
  -> 코스 생성 및 보호자 메시지 생성
  -> 일정 저장 / 공유 / 알림 / 지도
  -> 관리자 콘솔 / 파트너 콘솔 / 리포트 대시보드
```

### 기술 후보

| 영역 | 후보 |
| --- | --- |
| 프론트엔드 | React/Next.js, React Native, Flutter |
| 백엔드 | FastAPI, NestJS |
| 데이터베이스 | PostgreSQL, Redis |
| AI/RAG | OpenAI 모델, Embedding, Vector DB, Hybrid Search |
| 지도 | Kakao Map, Naver Map, Google Maps |
| 데이터 파이프라인 | scheduled jobs, Airflow류 워크플로우 |
| 인프라 | Cloud Run, ECS, Kubernetes, Object Storage |
| 보안 | 환경변수 관리, 키 관리, 접근제어, 로그 마스킹 |

선택 기준:

- 빠르게 출시할 수 있는가
- 운영자가 데이터를 갱신하기 쉬운가
- 개인정보와 API 키를 안전하게 관리할 수 있는가
- RAG 근거와 점수 산정 과정을 추적할 수 있는가
- 지도·공공데이터·제휴 API 변경에 대응하기 쉬운가

## 13. 안전·개인정보·법적 원칙

서비스는 의료 서비스가 아니다. 사용자의 건강 상태를 진단하거나 이동 가능성을 보장하지 않는다.

### 금지 표현과 대체 표현

| 금지 표현 | 대체 표현 |
| --- | --- |
| 휠체어도 100% 가능합니다. | 휠체어 접근 가능 여부는 현장 조건에 따라 달라질 수 있습니다. |
| 누구나 문제없이 갈 수 있습니다. | 방문 전 공식 정보와 현장 상황을 확인해 주세요. |
| 암환자에게 좋습니다. | 체력이 약한 여행자가 편안히 머물 수 있도록 이동 부담을 줄인 코스입니다. |
| 무조건 추천합니다. | 사용자의 이동 조건과 당일 상황에 따라 적합도가 달라질 수 있습니다. |
| 치료 효과가 있습니다. | 정서적으로 편안한 여행 경험을 돕는 코스입니다. |

### 개인정보 원칙

- 이름, 연락처, 주민등록번호, 병원명, 상세 진단명을 요구하지 않는다.
- 건강 상태는 이동 조건과 피해야 할 요소 수준으로만 해석한다.
- 사용자 입력 원문 저장은 기본값으로 피한다.
- 저장이 필요하면 비식별 요약만 저장한다.
- 보호자 공유 링크에는 만료 시간, 접근 권한, 민감 정보 제외 옵션을 둔다.
- 운영 로그에는 민감정보와 API 키가 남지 않게 한다.

## 17. 새 GPT 기획안 반영 판단

### 채택

- 상용 플랫폼 포지셔닝
- B2C/B2B/공공 수익모델
- 파트너 콘솔, 관리자 콘솔, 공공 리포트 구조
- Phase 1~4 로드맵
- KPI 체계
- 보호자 공유 링크의 만료·권한·민감정보 제외 옵션
- 접근성 리뷰를 조건형으로 수집하는 방향

### 수정 채택

- `전문가 검수 플랜`은 `운영자 재확인 플랜`으로 표현한다.
- 기술스택은 확정이 아니라 후보로 둔다.
- `안심 가능한 여행`은 외부 설명에서는 사용 가능하지만 결과 화면에서는 보장 표현처럼 보이지 않게 제한한다.
- 새 기획안의 장소 스키마는 현재 JSON Schema와 맞지 않으므로 참고용으로만 둔다.

### 채택하지 않음

- 장소 스키마를 새 문서의 부록 형식으로 교체하지 않는다.
- LLM이 점수를 직접 산정하는 구조로 가지 않는다.
- 전체 카탈로그 장소를 접근성 검증 장소처럼 취급하지 않는다.
- 초기부터 예약·결제·복잡한 계정 기능을 1차 출시 범위에 넣지 않는다.
