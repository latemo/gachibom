1. 현재 결론

제주의마음 접근성 여행 인공지능은 서비스화를 전제로 데이터 검수, 공공데이터 수령, 로드뷰 이미지 검증, 운영 준비 게이트를 구축하는 방향으로 정리됐다.

현재 자동 점검 기준은 전체 공개 차단 상태다. 기본 장소 데이터는 제한 공개 기준을 상당 부분 충족하고, 우선 검수 샘플 102장은 모두 확보됐다. 다만 서비스 시드 전체 원본 1,023장 중 70장이 제공 서버 오류로 남아 있어 전체 공개 승격은 아직 차단 상태다.
최우선 작업은 data/roadview_provider_404_image_report.json의 70건을 근거로 제공기관에 원본 복구 또는 대체 원본 수령을 요청하는 것이다. 확보된 102장 우선 샘플로 시각 검수는 시작할 수 있다.

2. 주요 기획 방향

  • 단순 데모가 아니라 상용 서비스 기준으로 진행한다.
  • 사용자 상황별 추천을 전제로 한다.
  • 암환자, 회복기 여행자, 고령자, 휠체어 사용자, 보호자 동반 여행자, 음식 제한 사용자, 혼잡 민감 사용자 등 상황별로 추천 장소와 제외 장소가 달라질 수 있도록 데이터 구조와 정책을 잡았다.
  • 식당, 음식 중심 장소, 실내 장소, 휴식 장소, 문화시설, 숲길, 시장 등 다양한 장소 유형을 수용한다.
  • 제주 전체 관광지와 식당 등을 장기적으로 확장하되, 초기 공개에는 검증 가능한 장소부터 단계적으로 반영한다.

3. 여행지 추천 및 점수 집계 방식

추천은 좋은 관광지를 홍보하는 방식이 아니라, 사용자가 입력한 제약과 장소별 접근성 카드가 얼마나 맞는지 계산하는 방식으로 설계했다. 진단명은 판단 기준이 아니며, 사용자가 직접 입력한 음식 제한, 이동 부담, 혼잡 회피, 날씨 민감 같은 여행 제약만 반영한다.

사용자 조건 정리

입력 묶음 예시 추천 처리
traveler_type 휠체어 사용자, 고령자, 회복기 여행자, 보호자 동반 이동 부담, 필수 편의시설, 확인 항목의 우선순위를 조정
mobility_conditions 긴 걷기 어려움, 체력 저하, 날씨 민감, 혼잡 회피 도보 부담, 야외 체류, 혼잡 가능성을 감점 또는 확인 필요로 반영

추천 처리 순서

  1. 검증 가능한 접근성 장소 카드에서 후보를 만든다.
  2. 비공개, 차단, 폐기 상태인 장소는 기본 추천 후보에서 제외한다.
  3. 음식 제한, 혼잡 민감, 날씨 민감, 감각 민감 등 상황별 제외·감점 규칙을 먼저 적용한다.
  4. 제외되지 않은 장소에 100점 기준 적합도 점수를 계산한다.
  5. 차단되지 않은 장소를 점수와 신뢰도 순으로 정렬한다.
  6. 상위 장소를 최대 4곳까지 묶어 코스를 만든다.
  7. 결과에는 점수, 추천 이유, 감점 이유, 방문 전 확인 항목, 출처, 안전 문구를 함께 표시한다.

상황별 제외와 감점

  • 음식 제한 사용자는 식당과 음식 중심 시장을 제외하고, 카페는 감점한다.
  • 혼잡·감염 민감 사용자는 시장, 쇼핑, 카페, 혼잡 가능 태그 장소를 감점한다.
  • 날씨 민감 사용자는 해안, 오름, 장시간 야외 장소를 감점한다.
  • 감각 민감 사용자는 강한 조명, 소리, 어두운 전시 등 자극 요소를 확인 필요로 둔다.

코스 점수 집계

rank_places가 장소별 점수를 계산하고 차단 장소를 제외한 뒤 정렬한다. build_recommendation_result는 상위 장소를 최대 4곳까지 선택하며, 코스 총점은 선택된 장소 점수의 평균이다. 코스 신뢰도는 선택된 장소 중 가장 낮은 신뢰도를 따른다.
  • 등급은 평균 점수 기준으로 A/B/C/D/F를 다시 계산한다.
  • 추천 이유, 감점 이유, 확인 필요 항목, 출처는 선택된 장소들의 값을 중복 제거해 합친다.
  • 조건에 맞는 장소가 없으면 무리한 추천을 하지 않고 추천 보류 결과를 만든다.

  • data/schemas/recommendation_result.schema.json: 추천 결과 출력 형식 검증
  • docs/jeju_maeum_scoring_policy.md: 100점 배점, 강제 감점, 차단 규칙

  • tests/test_scoring.py: 주요 추천·점수 규칙 테스트

4. 현재까지 개발한 근거

기획 근거

  • 사용자가 데모가 아니라 서비스화를 요구했기 때문에, 화면보다 데이터 품질, 출처, 검수, 운영 게이트를 먼저 잡았다.
  • 암환자 예시처럼 사용자 상황에 따라 식당을 제외해야 하는 요구가 있어, 진단명이 아니라 여행 제약으로 처리하는 상황별 추천 정책을 만들었다.
  • 제주 전체 관광지와 식당까지 확장하려는 방향이 있으므로, 초기 데이터만 하드코딩하지 않고 전체 장소 카탈로그와 접근성 카드를 분리했다.
  • 추천 결과가 과장되면 안 되므로 추천보다 근거, 가능보다 부담, 의료 판단 금지 원칙을 문서화했다.

데이터 근거

  • 공공데이터 15109149, 15110209, 15109158, 15109153을 검토해 공개 연계, 이미지 원본, 이미지 메타데이터, 시설 현황의 역할을 분리했다.
  • 기본 장소 카드 43개는 스키마 검증 대상이며, 출처, 안전 메모, 접근성 상태, 운영자 메모를 포함한다.
  • 로드뷰 기반 서비스 시드 17곳은 공개 전 비공개로 유지하고, 이미지 수령과 시각 검수 전에는 활성 후보로 승격하지 않는다.
  • 로드뷰 이미지 원본 1,023장을 요청 대상으로 만들고, 그중 102장을 우선 검수 샘플로 분리했다.
  • 공간정보 이미지 주소 패턴으로 원본 직접 내려받기를 수행해 953장을 확보했고, 서버 404 70건을 별도 리포트로 분리했다.

기술·운영 근거

  • jsonschema 검증을 전제로 장소 카드, 추천 결과, 운영 리포트, 로드뷰 검수 산출물의 스키마를 만들었다.
  • 공공데이터 수령 전에도 진행 상태를 판단할 수 있도록 데이터 요청 추적대장, 이미지 수령 리포트, 서비스 시드 게이트, 운영 준비 리포트를 만들었다.
  • 전체 테스트 76개가 통과해 현재 기준의 스키마, 점수화, 공공데이터 변환, 로드뷰 검수 흐름이 깨지지 않는지 확인했다.
  • 현재 전체 공개 차단 상태는 실패가 아니라, 상용 공개 전 이미지 수령과 시각 검수가 아직 끝나지 않았다는 운영 게이트 결과다.

11. 주의사항

  • 이미지 원본은 저장소에 직접 커밋하지 않는다.
  • .env 값이나 다른 외부 서비스키는 응답, 문서, 코드에 노출하지 않는다.
  • 로드뷰 이미지 검수 전에는 서비스 시드 17곳을 공개 활성 카드로 승격하지 않는다.

  • 공식 출처 없는 장소나 안전 메모 없는 장소는 상용 추천 결과에 노출하지 않는다.
  • 추천 점수는 여행 가능성 보장이 아니라 판단 보조 신호로 표시한다.
  • 암환자, 회복기, 고령자 같은 표현은 의료 판단이 아니라 사용자가 직접 입력한 여행 제약을 정리하는 용도로만 사용한다.
